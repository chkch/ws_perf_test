# WebSocket 性能压测工具

一个专为大规模、长时间 WebSocket 连接测试设计的 Node.js 压测工具。

**支持能力**：单机 5 万并发连接，7x24 小时稳定运行。

## v3.0 优化特性

| 类别 | 优化项 | 说明 |
|------|--------|------|
| **性能** | A1 百分位数计算优化 | 二分插入 O(log n) 替代排序 O(n log n) |
| | A2 Buffer 池复用 | 减少 GC 压力 |
| | A3 连接预热模式 | 先建连再开始测试统计 |
| | A4 并发创建限制 | 避免瞬时过载 |
| **监控** | C1 连接时间百分位 | P50/P95/P99 建连时间 |
| | C2 吞吐量 TPS 统计 | 峰值追踪 |
| | C3 FD 文件描述符监控 | 预警系统限制 |
| | C4 连接状态分布 | 实时显示各状态连接数 |
| | C5 TTFB 首字节时间 | 首条消息延迟统计 |
| **稳定性** | E1 连接超时优化 | 独立超时计时器，避免 pending 堆积 |
| | E2 内存泄漏检测 | 定期检查连接 Map |
| | E3 优雅降级 | 资源紧张时自动停止新建连接 |
| | E4 DNS 缓存 | 减少 DNS 查询开销 |

## 功能特性

- **大规模连接** - 单机支持 5 万+ 并发 WebSocket 连接
- **长时间稳定** - 针对 7x24 小时测试优化，防止内存泄漏
- **实时监控面板** - 连接状态、消息吞吐、延迟、内存使用一目了然
- **智能重连** - 指数退避重连机制，防止重连风暴
- **延迟统计** - P50/P95/P99 百分位延迟，基于 Ping/Pong RTT
- **TTFB 统计** - 首字节时间，测量首条消息延迟
- **优雅降级** - 内存/FD 资源紧张时自动停止新建连接
- **优雅退出** - Ctrl+C 正常关闭所有连接，通知服务端
- **结果导出** - 支持 JSON/CSV 格式导出测试报告
- **WSS 支持** - 支持自定义 Headers、跳过证书验证

## 安装

```bash
git clone https://github.com/chkch/ws_perf_test.git
cd ws_perf_test
npm uninstall ws-perf-test -g && npm install && npm install . -g
chmod +x /usr/local/bin/ws-perf-test
```

## 快速开始

```bash
# 基础测试
ws-perf-test ws://localhost:9502 -c 1000 -d 60

# 预热模式测试
ws-perf-test ws://localhost:9502 -c 5000 --warmup -d 120

# 查看帮助
ws-perf-test --help
```

## 命令行参数

### 连接选项

| 参数 | 简写 | 默认值 | 说明 |
|------|------|--------|------|
| `--connections` | `-c` | 10000 | 总连接数 |
| `--batch` | `-b` | 500 | 每批建立的连接数 |
| `--interval` | `-i` | 1000 | 批次间隔（毫秒） |
| `--reconnect` | `-r` | false | 启用断线重连 |
| `--reconnectDelay` | - | 3000 | 重连基础延迟（毫秒） |
| `--maxReconnect` | - | 10 | 单连接最大重连次数 |
| `--rateLimit` | - | 0 | 每秒最大新建连接数，0=不限 |
| `--concurrentLimit` | - | 100 | 并发建连限制 [A4] |
| `--connectTimeout` | - | 15000 | 连接超时（毫秒）[E1] |

### 消息选项

| 参数 | 简写 | 默认值 | 说明 |
|------|------|--------|------|
| `--pingInterval` | - | 30000 | Ping 心跳间隔（毫秒） |
| `--data` | - | null | 订阅消息 JSON 文件（连接后只发送一次） |

### 测试选项

| 参数 | 简写 | 默认值 | 说明 |
|------|------|--------|------|
| `--duration` | `-d` | 0 | 测试时长（秒），0=无限 |
| `--log` | `-l` | null | 日志输出文件（包含详细错误信息） |
| `--warmup` | - | false | 预热模式：先建连再开始测试 [A3] |
| `--export` | - | null | 结果导出文件 |
| `--exportFormat` | - | json | 导出格式: json, csv |

### 稳定性选项

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--killZombie` | false | 自动终止僵尸连接 |
| `--zombieTimeout` | 120000 | 僵尸判定超时（毫秒） |
| `--gcInterval` | 60000 | 手动 GC 间隔（毫秒） |
| `--maxMemoryPercent` | 85 | 内存使用上限百分比，超过则降级 [E3] |
| `--maxFdPercent` | 80 | FD 使用上限百分比，超过则降级 [E3] |

### WSS/安全选项

| 参数 | 说明 |
|------|------|
| `--insecure` | 跳过 WSS 证书验证 |
| `--headers` | 自定义 HTTP Headers（JSON 格式） |
| `--subprotocol` | WebSocket 子协议 |

## 5 万连接 7x24 小时稳定性测试

### 前置条件

#### 1. 系统配置（客户端机器）

```bash
# 查看当前文件描述符限制
ulimit -n

# 临时增加限制（当前会话）
ulimit -n 65535

# 永久修改（需要 root）
# 编辑 /etc/security/limits.conf 添加：
# *         soft    nofile      65535
# *         hard    nofile      65535
```

#### 2. TCP 参数优化（Linux）

```bash
# 编辑 /etc/sysctl.conf 添加：
net.ipv4.tcp_tw_reuse = 1
net.ipv4.ip_local_port_range = 1024 65535
net.core.somaxconn = 65535
net.ipv4.tcp_max_syn_backlog = 65535

# 应用配置
sudo sysctl -p
```

#### 3. macOS 配置

```bash
sudo launchctl limit maxfiles 65535 65535
```

### 推荐启动命令

```bash
# 5 万连接 7x24 小时稳定性测试（带优雅降级）
ulimit -n 100000

node --expose-gc --max-old-space-size=4096 \
  ws-perf-test ws://your-server:9502 \
  -c 50000 -b 500 -i 500 \
  --data subscribe.json \
  -d 604800 \
  -l stability_50k.log \
  --export result.json \
  --killZombie \
  --maxMemoryPercent 80 \
  --concurrentLimit 200
```

**参数说明**：

| 参数 | 值 | 说明 |
|------|------|------|
| `-c` | 50000 | 5 万连接 |
| `-b` | 500 | 每批 500 个，约 100 秒建立完成 |
| `-i` | 500 | 批次间隔 500ms |
| `--data` | subscribe.json | 订阅消息文件 |
| `-d` | 604800 | 运行 7 天 |
| `-l` | stability_50k.log | 日志持久化 |
| `--export` | result.json | 导出测试结果 |
| `--killZombie` | - | 自动终止僵尸连接 |
| `--maxMemoryPercent` | 80 | 内存超 80% 时降级 |
| `--concurrentLimit` | 200 | 并发建连限制 |

### 使用 nohup 后台运行

```bash
# 12小时稳定性压测
nohup ws-perf-test ws://xxxx/v1/public \
  -c 50000 -b 500 -i 1000 \
  --data subscribe.json \
  -d 43200 \
  -l stability_50k.log \
  --export result.json \
  --maxMemoryPercent 80 \
  > /dev/null 2>&1 &
```

### 使用 tmux 后台运行（推荐）

```bash
# 创建 tmux 会话
tmux new -s ws_test

# 运行测试
ulimit -n 100000
ws-perf-test ws://your-server:9502 \
  -c 50000 -b 500 -i 500 \
  --data subscribe.json \
  -d 604800 \
  -l stability_50k.log \
  --export result.json \
  --killZombie

# 分离会话：按下 Ctrl+b，然后按 d [程序在后台运行]
# 查看会话： tmux ls
# 重新连接：tmux attach -t ws_test
# 清除所有会话：tmux kill-server
```

## 常见问题

### Q1: 报错 "EMFILE: too many open files"

```bash
ulimit -n 100000
```

### Q2: 连接数卡在某个值上不去

```bash
# Linux: 扩大端口范围
sudo sysctl -w net.ipv4.ip_local_port_range="1024 65535"
sudo sysctl -w net.ipv4.tcp_tw_reuse=1
```

### Q3: 内存持续增长

```bash
# 启用手动 GC + 降级保护
node --expose-gc --max-old-space-size=4096 src/ws_perf_test.js \
  --maxMemoryPercent 80 ...
```

### Q4: 连接建立很慢

```bash
# 增加并发建连限制
ws-perf-test ... --concurrentLimit 200
```

### Q5: 如何使用预热模式？

预热模式会先建立所有连接，再开始统计测试数据，适合需要精确测量稳态性能的场景：

```bash
ws-perf-test ws://server:9502 -c 10000 --warmup -d 300
```

### Q6: 如何安全停止？

按 `Ctrl+C`，工具会：
1. 停止创建新连接
2. 批量关闭所有连接（发送 close frame 通知服务端）
3. 打印最终测试报告
4. 导出结果文件（如配置了 --export）
